# drone-H743 Project Skill

## Scope

Use this skill for the current repository `D:\stm32hal\drone-H743`.

This is an STM32H743 firmware project generated by STM32CubeMX and organized around FreeRTOS, board bring-up, sensor drivers, flash storage, UART/Wi-Fi telemetry, and flight-control application code.

## CubeMX Boundary

Treat `drone-H743.ioc` as the hardware configuration source of truth.

CubeMX-managed files include:

- `Core/Src/main.c`
- `Core/Src/freertos.c`
- `Core/Src/gpio.c`
- `Core/Src/dma.c`
- `Core/Src/spi.c`
- `Core/Src/i2c.c`
- `Core/Src/usart.c`
- `Core/Src/tim.c`
- `Core/Src/stm32h7xx_hal_msp.c`
- matching headers in `Core/Inc/*`
- `USB_DEVICE/*`

If a task changes pins, peripheral instances, DMA, NVIC, clocks, or FreeRTOS objects that CubeMX owns, first tell the user exactly what to change in CubeMX and wait for regenerated code before repairing hand-written layers.

Hand-written project layers are mainly:

- `App/*`
- `BSP/*`
- `Driver/*`
- `tools/*`
- `tests/*`

## Layering Rule

Current intended layering:

```text
App / Services
  -> Driver
  -> BSP / Board binding
  -> HAL / MCU
```

Meaning:

- `App/*`: application behavior, diagnostics, commands, storage policy, control flow.
- `App/*_service.*`: application-facing service boundary when an App feature needs shared driver access, locking, or policy.
- `Driver/*`: portable chip/protocol drivers, named after chips or device classes, not after the board.
- `BSP/*`: board-specific bus binding, GPIO/CS, DMA callback registration, cache policy wiring, and board resource ownership.
- `Core/*`: CubeMX/HAL-generated MCU configuration.

Do not put chip protocols into BSP names. BSP is below Driver in this project.

## Current FLASH Layer

The external NOR flash is currently organized as:

```text
App/Inc/app_flash_service.h
App/Src/app_flash_service.c
  -> Driver/Inc/drv_gd25q32.h
  -> Driver/Src/drv_gd25q32.c
  -> BSP/Inc/bsp_flash_bus.h
  -> BSP/Src/bsp_flash_bus.c
  -> BSP board SPI1/CS/DMA/cache binding
```

Key rules:

- App code should call `APP_FlashService_*`, not `BSP_FLASH_*`.
- GD25Q32 chip commands belong in `DRV_GD25Q32_*`.
- `BSP_FlashBus_*` only owns bus binding, locking, DMA callback registration, and board-specific cache hooks.
- Legacy `bsp_gd25q32.*`, `bsp_flash.*`, `drv_flash.*`, and `BSP_FLASH_*` names should not come back.

Known hardware facts:

- Flash chip: GD25Q32 / JEDEC `C8 40 16`
- Flash bus: SPI1, `FLASH_CS=PA4`, `SCK=PA5`, `MISO=PA6`, `MOSI=PA7`
- Current专项测试 commands: `RTOS?`, `FLASH?`, `FLASH VERIFY <addr> <len>`, `FLASH BENCH READ <addr> <len> <loops>`
- Host test tool: `python tools/flash_diag_test.py --serial COMx --baud 115200 --final-rtos`

## STM32H743 DMA And Cache

Always consider H7 memory domains and D-cache when DMA is involved.

Rules:

- Do not assume DMA can access DTCM.
- DMA buffers should be in a DMA-reachable region and aligned when cacheable.
- If cacheable DMA buffers are used, clean before peripheral reads memory and invalidate after peripheral writes memory.
- For this repo, flash DMA cache maintenance is wired through the `DRV_GD25Q32_Bus` callbacks from BSP.
- Before linker, MPU, or section changes, inspect `STM32H743XX_FLASH.ld` and generated startup/config files.

## FreeRTOS Rules

Keep ISR code short and wake tasks with RTOS primitives.

Typical choices:

- ISR wakeup: task notification
- shared SPI/UART bus: mutex
- command/event fan-in: queue or event flags
- byte stream: stream/message buffer
- latest-state data: double buffer or publish/subscribe style cache

When adding CubeMX-created tasks, ask the user to configure them in CubeMX first, then inspect generated `Core/Src/freertos.c` and connect the task body in `App/*`.

## Validation Rule

For architecture and naming changes, add or update tests in `tests/*` before or with the implementation.

Current validation commands:

```powershell
python -m pytest tests\test_flash_layering.py tests\test_flash_bdd.py -q
cmake --build --preset Debug
python -m py_compile tools\flash_diag_test.py
```

Use BDD-style tests to describe expected architecture scenarios, and TDD-style tests to prevent old names or forbidden dependencies from returning.

## Current Services And Background Rule

`Services/*` contains no-task service APIs. A service owns data rules and synchronous operations; it is not an App task.

Current service split:

- `Services/Inc/svc_param.h`, `Services/Src/svc_param.c`: RAM parameter mirror, boot-time parameter load from FLASH, dirty flag, slow-save request.

There is currently only one service: Param. `backgroundTask` is an App task, not a service.

Runtime slow work goes through the background task boundary:

```text
other App
  -> backgroundReqQueue / APP_BackgroundRequest
  -> backgroundTask / APP_Task_Background_Step()
  -> Param API or APP_FlashService_* synchronous call
  -> backgroundRespQueue / APP_BackgroundResponse
```

Do not restore `storageReqQueue`, `storageRespQueue`, `storageTaskHandle`, `APP_Storage`, `APP_STORAGE_*`, `flashReqQueueHandle`, `flashTaskHandle`, `APP_FLASH_Request`, or `APP_FLASH_REQ_*`.

`Param` is a service, not an App. `backgroundTask` is the low-priority App task that executes slow operations by calling Param APIs. `APP_FlashService_*` remains a synchronous lower-level service used by Param, boot-time load, and maintenance diagnostics.
