# STM32H743 Memory And Domain Notes For This Project

这个参考文件只在任务涉及性能、任务栈、DMA、cache、一致性或大 buffer 规划时读取。

## 先记住的结论

- H743 不是简单的单块 RAM 架构
- 项目里要把“CPU 热代码”“CPU 私有高速数据”“大通用数据”“DMA/外设侧数据”分开考虑
- 真正需要改 linker、MPU、section 之前，先确认当前 `.ioc` 和 `STM32H743XX_FLASH.ld`

## 当前工程 `.ioc` 已暴露的主要内存区

- `ITCMRAM` at `0x00000000`, size `64 KB`
- `DTCMRAM` at `0x20000000`, size `128 KB`
- `RAM` at `0x24000000`, size `512 KB`
- `RAM_D2` at `0x30000000`, size `288 KB`
- `RAM_D3` at `0x38000000`, size `64 KB`

## 项目内推荐的四区分工

### 1. ITCM

适合：

- 非常热的短函数
- 对抖动敏感的核心控制代码
- 小而频繁的数学热点

不适合：

- 大段普通业务代码
- 还没有测过热点就先搬进去的代码

原则：

- 先测热点，再决定是否迁移
- 没有性能证据时，先留在默认代码区

### 2. DTCM

适合：

- 只被 CPU 使用的数据
- 高频访问的小型结构体
- 任务栈
- 中间计算缓冲

谨慎点：

- 不要把要给 DMA 用的 buffer 默认放这里
- 当任务栈很大时，先考虑拆任务，再考虑继续堆栈

### 3. D1 / AXI SRAM

适合：

- 较大的通用 buffer
- 大型数据结构
- 吞吐优先的工作区
- 需要带宽的共享数据

谨慎点：

- 若 buffer 会被 DMA 使用，要同步考虑 cache 一致性
- 若要开大 cache 优化，需要一起看 MPU 策略

### 4. D2 / D3 SRAM

适合：

- 更贴近外设交互的数据
- DMA buffer
- 低速共享状态
- 希望和主计算区分开的常驻数据

建议：

- SPI/I2C/UART/USB 这类 I/O 的 DMA buffer，优先从这里或可验证的 DMA 可达 RAM 中规划
- D3 可留给低速、常驻、小体量状态

## FreeRTOS 相关建议

- 大任务栈不是默认解法，优先拆职责
- 一个任务既做采样、解析、格式化、显示，通常是坏信号
- 大局部数组、`printf` 风格格式化、协议拼包，都会把栈推高
- 修改任务栈之前，先说明：
  - 为何栈会变大
  - 能否把大对象改成静态区或专用 buffer
  - 能否拆成生产者/消费者两个任务

## DMA 和 Cache 的最小检查清单

只要涉及 DMA buffer，就至少检查：

1. buffer 在哪块 RAM
2. 这块 RAM 对当前 DMA 主设备是否可达
3. 这块 RAM 当前是否 cacheable
4. 如果 cacheable，发送前是否 clean，接收后是否 invalidate
5. 有没有更简单的方案，比如把 DMA buffer 放进单独的非缓存区

## 需要显式提醒用户的情形

遇到下面情况时，要主动写出 H7 资源建议，而不是只给代码：

- 新增高频控制算法
- 新增传感器 DMA 采样链
- 新增屏幕刷新或大块 SPI 数据搬运
- 新增 USB 高吞吐路径
- 任务栈明显上涨
- 想通过“加 cache”直接提速
- 想把外部项目的大模块原样移植进来

## 推荐的输出模板

当任务明显受 H7 架构影响时，在回答里加一个短段落：

`H7 资源建议`

- `代码区`: 说明是否需要考虑 ITCM
- `任务栈/控制数据`: 说明是否更适合 DTCM
- `大 buffer`: 说明是否更适合 D1 AXI SRAM
- `DMA buffer`: 说明是否更适合 D2/D3，并指出 cache 风险
- `是否需要 CubeMX 或 linker/MPU 配合`: 写清楚要不要先停下来改配置
